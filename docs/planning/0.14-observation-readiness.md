# 0.14 Observation Readiness Plan

## Purpose

`0.14.x` should not implement observation-learning directly. It should make observation meaningful later by establishing world time, daypart, darkness, visibility and light-source rules.

## Design Rule

`0.14` makes **seeing / not seeing** systemic.

`0.15` makes **watching / learning** systemic.

## 0.14 Foundations Required Before `/observe`

1. Stable world-time state.
2. A daypart helper for dawn/day/dusk/night.
3. A shared visibility service.
4. Darkness reducing location detail.
5. Darkness reducing nearby beings, tracks and ground objects.
6. Active light sources revealing what darkness hides.
7. Firewood/foraging content that creates natural future observation targets.

## Future-Facing Visibility Contract

The visibility service should eventually let later systems ask:

```ts
getVisibilityContext({
  viewerPlayerId,
  locationId,
  targetLocationId,
})
```

A future result may look like this:

```ts
{
  daypart: "dawn" | "day" | "dusk" | "night";
  hasLocalLight: boolean;
  visibilityTier: "clear" | "dim" | "dark";
  hidesDetails: boolean;
  canReadTracks: boolean;
  canSeeNearbyBeings: boolean;
  canSeeGroundObjects: boolean;
}
```

This shape is a guide, not a required exact API. Keep the first implementation tiny.

## Avoid During Early 0.14

- Do not introduce observation XP.
- Do not introduce `PlayerSkill` just for future observation.
- Do not add `/observe` / `Спостерігати` yet.
- Do not reveal hidden presence or invisible spirits.
- Do not make darkness feel like a technical filter.
