# 0.14.7 - Post-Merge Herald Render Operations Docs - 12026-06-01

This release documents the intended post-merge Render setup for the main
Chornolis game bot and the Boundary Mark Chancery / Канцелярія Межового Знаку,
and hardens Herald news recovery from the durable publication outbox.

## Added

- Added `docs/ops/render-services.md` as the canonical post-merge two-service
  Render setup:
  - one repository;
  - one `main` branch;
  - separate main game and Herald Render services;
  - different Start Commands;
  - separate bot tokens and env ownership.
- Added explicit duplicate-polling warnings for `HERALD_BOT_TOKEN`.
- Added the current Prisma migration rule for the free Render Web Service
  Herald build command.
- Added status-page expectations for showing the Herald heartbeat and
  HeraldPublication queue/published counts without exposing secrets.
- Added HeraldPublication snapshot fields for rendered text, source date and
  source version metadata where available.
- Added admin-only `/show_publication <id>` to display a saved publication
  snapshot even if the source news entry has since been removed.
- Added admin-only `/repost_publication <id>` to explicitly repost a saved
  snapshot as an archive/repost message. This creates a new Telegram message
  and does not pretend to restore the original Telegram timestamp.
- Added admin-only `/list_publications` and `/mark_publication_deleted <id>` for
  manual recovery when a channel post is deleted outside the bot. Telegram does
  not reliably notify the Herald about ordinary manual channel deletions.
- Added admin-only `/info_full` for a fuller Herald character record while
  keeping public `/info` limited to safe public/diegetic character information.

## Changed

- Updated `docs/dev/render_deploy.md` to distinguish the main game service's
  seed-aware build/deploy flow from the Herald Web Service flow.
- Updated `docs/ops/herald-bot.md` to point at the post-merge `main` branch
  deployment model.
- Updated `HERALD-001` so the future embedded-mode icebox item starts from the
  current standalone two-service baseline.
- Updated `WEB-002` so the independent status/deploy-visibility backlog item
  accounts for the new game-hosted Herald heartbeat/status overview.
- Latest-news and backfill queueing still deduplicate by `contentHash`, but
  now store and reuse the rendered snapshot text for later recovery.
- Herald `/info` is now public and reply-aware: plain `/info` resolves the
  caller, while replying with `/info` resolves the replied-to Telegram user when
  Telegram exposes that sender. Anonymous/channel-origin messages get a visible
  atmospheric refusal instead of a silent failure or admin-gate notice.

## Deferred

- Embedded Herald mode remains icebox work.
- Independent out-of-band status hosting remains backlog work.
- `HERALD_STARTUP_NOTICE_THREAD_ID` is documented as reserved until runtime
  support exists.

## Risks / Rollback

- Existing queued/published rows from before this migration may not have a
  rendered snapshot yet, but they still retain `title` and `body`; queueing the
  same content again can fill missing snapshot metadata when the `contentHash`
  matches.
- Manual deletion markers are advisory DB state only; the Herald does not and
  cannot reliably prove that a Telegram channel message is gone.
- `/repost_publication` is explicit admin action only; it can create duplicate
  channel posts if used intentionally.
- Operational risk is mostly stale docs: if Render commands change later, update
  `docs/ops/render-services.md`, `docs/dev/render_deploy.md` and
  `docs/ops/herald-bot.md` together.
- Rollback: revert this patch to return to the `0.14.6` documentation and
  Herald publication behavior; the snapshot migration is additive and can be
  left in place if already deployed.
