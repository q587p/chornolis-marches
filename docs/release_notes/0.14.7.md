# 0.14.7 - Post-Merge Herald Render Operations Docs - 12026-06-01

This release documents the intended post-merge Render setup for the main
Chornolis game bot and the Boundary Mark Chancery / Канцелярія Межового Знаку.

No runtime behavior changes are intended in this patch.

## Added

- Added `docs/ops/render-services.md` as the canonical post-merge two-service
  Render setup:
  - one repository;
  - one `main` branch;
  - separate main game and Herald Render services;
  - different Start Commands;
  - separate bot tokens and env ownership.
- Added explicit duplicate-polling warnings for `HERALD_BOT_TOKEN`.
- Added the current Prisma migration rule for the free Render Web Service
  Herald build command.
- Added status-page expectations for showing the Herald heartbeat and
  HeraldPublication queue/published counts without exposing secrets.

## Changed

- Updated `docs/dev/render_deploy.md` to distinguish the main game service's
  seed-aware build/deploy flow from the Herald Web Service flow.
- Updated `docs/ops/herald-bot.md` to point at the post-merge `main` branch
  deployment model.
- Updated `HERALD-001` so the future embedded-mode icebox item starts from the
  current standalone two-service baseline.
- Updated `WEB-002` so the independent status/deploy-visibility backlog item
  accounts for the new game-hosted Herald heartbeat/status overview.

## Deferred

- Embedded Herald mode remains icebox work.
- Independent out-of-band status hosting remains backlog work.
- `HERALD_STARTUP_NOTICE_THREAD_ID` is documented as reserved until runtime
  support exists.

## Validation

- Ran `npm run planning:export`.
- Ran `npm run build`.
- Ran `npm test`.
- Ran `git diff --check`.

## Risks / Rollback

- Runtime risk should be low because this patch is documentation and planning
  metadata only.
- Operational risk is mostly stale docs: if Render commands change later, update
  `docs/ops/render-services.md`, `docs/dev/render_deploy.md` and
  `docs/ops/herald-bot.md` together.
- Rollback: revert this docs patch to return to the `0.14.6` documentation
  state; no database or runtime rollback is expected.
