# 0.15.31 - Follow route memory stabilization

Date: 12026-06-04

## Summary

This release stabilizes the first follow-aware route-memory slice before any
future follow movement work. Follow intent can still help a player notice a
watched target's ordinary visible movement, but repeated movement now has
narrow cooldowns so the same route does not keep spamming proactive hints or
silent learning progress.

This remains attention context, not automatic travel. The follower is never
moved, queued, teleported or grouped by this system.

## Player-Facing Changes

- Repeated followed movement through the same source/direction is less noisy.
- `/me` now marks a followed target that is no longer visible nearby as
  `останній помічений`, so the status reads as last-known attention rather than
  a guaranteed live lock.
- Darkness and hidden water-word passage movement still do not reveal ordinary
  direction memory.

## Technical Notes

- Added `cooldownKey` and `learningKey` markers to follow route-memory
  `WorldEvent` descriptions.
- Visible follow route-memory hints use a short per-follower/target/source/
  direction cooldown.
- Hidden route-memory hints use a longer per-follower/target/source cooldown.
- Silent `tracking` / `observation` / `followed_movement` progress uses a
  separate per-follower/target cooldown to avoid rapid back-and-forth farming.
- Follow route memory still uses the existing location-wide light snapshot via
  `visibilityRulesForLocation(...)`; no viewer-specific light refactor was
  added in this patch.
- No Prisma migration was added.

## Out Of Scope

- No automatic movement.
- No route replay.
- No group/party/travel membership.
- No companion or pet behavior.
- No hidden passage auto-repeat.
- No public skill UI or broad tracking modifier.
- No NPC schedule changes or new map content.

## Risks / Watchpoints

- Cooldowns use `WorldEvent` history, so aggressive pruning of recent follow
  route-memory events could make hints or silent progress repeat sooner.
- The light check is still location-wide. If future carried-light behavior needs
  viewer-specific precision, route memory should move to a viewer-aware
  visibility helper.
