# 0.15.7 - Player-made campfires - 12026-06-02

This patch turns `twigs` into the first player-made campfire loop.

## Summary

- Added player-made ordinary campfires built from `twigs ×5`.
- Added prepared unlit campfire state: the feature is visible, but does not provide light until lit from a carried burning torch.
- Added queued `BUILD_CAMPFIRE`, `DOUSE_CAMPFIRE` and `DISMANTLE_CAMPFIRE` actions.
- Added inventory and feature-inspection buttons for building, lighting, dousing and dismantling handmade campfires.
- Added wet `RIVER`/`SWAMP` handling: wet builds ask for confirmation and burn for a shorter duration once lit.
- Added a local cap of three active handmade ordinary campfires per location.
- Added focused tests for campfire helper rules and action costs.
- Added campfire system and implementation-planning docs.

## Player-facing

- `twigs` can now become a small prepared campfire instead of only extending an existing one.
- Prepared campfires can be lit with a burning torch.
- Active handmade campfires can be doused.
- Prepared or extinguished handmade campfires can be dismantled for partial `twigs` recovery.
- Wet places still allow campfire building, but warn first and do not keep heat as long.

## Risks / Rollback

- A schema migration adds three `WorldActionType` enum values. Deploy migration before running the new bot code.
- Campfire balance may need tuning after live play: five `twigs`, a three-fire local cap and partial recovery are intentionally conservative first values.
- Wet campfires shorten burn duration immediately after lighting; river/swamp survival pacing may feel harsher until tuned.
- PostgreSQL enum rollback is not a simple reverse of `ALTER TYPE ... ADD VALUE`. If player-made campfire actions need to be disabled after deploy, prefer reverting the code and leaving unused enum values in place unless a deliberate enum-rebuild migration is prepared.
