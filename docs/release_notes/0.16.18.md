# 0.16.18 — Action queue responsiveness

Date: 12026-06-09
Branch: `codex/0.16.18-action-queue-responsiveness`

## Summary

- Made identical active player `TRACK` actions idempotent, including the meaningful payload, so repeated tracking clicks do not stack duplicate running/queued searches.
- Applied the same narrow idempotency to `LOOK` actions.
- Answered `LOOK`/`TRACK` callback queries with a short early acknowledgement before queue DB work, then kept detailed queue feedback in the normal reply path.
- Removed full `recoverStamina(bot)` work from `processActionQueue(...)`.
- Added a guarded `recoveryLoop` that runs stamina/HP recovery on `RECOVERY_POLL_MS`, defaulting to 5 seconds.
- Added `slow:actionQueue.process` and `slow:recoveryLoop.pass` labels for short live validation.

## Technical Notes

- Deduplication is opt-in for observation actions only. Repeated `MOVE`, `SAY`, `ATTACK`, `GATHER` and other intentional repeated actions keep their existing queue behavior.
- The dedupe key includes the action type and stable JSON payload, so `/track fox` and `/track rabbit` remain distinct.
- Recovery behavior remains the existing recovery implementation; this PR only moves it out of the hot action-queue poll and gives it a guarded loop.
- `SLOW_COMMAND_LOG_MS=100` can be used during a live smoke to verify that action queue and recovery timings appear under separate labels.

## Risks / Rollback

- Recovery now follows the separate recovery loop cadence, so rest/sleep notifications may align to that loop rather than to each action queue poll.
- This is intentionally not a query-optimization rewrite of every recovery candidate path; it first removes recovery from the 100 ms queue hot loop.
- No Prisma schema, migrations, WorldEventMarker, sleep comfort, predator visibility, vertical movement, social alias, totem behavior or other 0.16.17 gameplay-surface changes.
- Rollback is a normal revert of the release commit.
