# 0.16.22 — Recovery loop pruning

Date: 12026-06-09
Branch: `codex/0.16.22-recovery-loop-pruning`

## Summary

- Pruned the creature half of the separate recovery loop so it no longer scans every non-gone creature every pass.
- Added recovery loop runtime snapshots for running state, last finish/error, player/creature phase timings and recovery counts.
- Extended guarded `/queueDebug` diagnostics with recovery timing/count lines beside the action and creature queue sections.
- Added focused recovery-loop tests for the runtime snapshot, debug report and creature scan guardrail.

## Technical Notes

- Creature recovery candidates are now limited to creatures below base stamina, creatures with a custom `staminaMax`, and currently queued/running creatures that may need `lastStaminaRegenAt` refreshed.
- Recovery still uses the existing `slow:recoveryLoop.pass` slow-log label and the existing overlap guard.
- The recovery loop still handles passive/rest/sleep stamina recovery, HP recovery, ordinary sleep auto-wake, regained-consciousness handling, rest completion prompts, tutorial idle/rest comments, creature passive stamina recovery and active actor recovery timestamp refreshes.

## Risks / Rollback

- This is a pruning/diagnostics pass, not a gameplay tuning change.
- No Prisma schema, migrations, action duration, stamina cost, fatigue threshold, HP/stamina formula, player queue, movement, combat, tracking, predator, sleep, social, totem or WorldEventMarker behavior changes.
- If live logs still show repeated `slow:recoveryLoop.pass`, inspect `/queueDebug` recovery counts and phase timings before widening the recovery scan again.
- Rollback is a normal revert of the release commit.
